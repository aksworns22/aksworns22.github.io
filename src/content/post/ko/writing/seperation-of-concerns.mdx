# 사라진 관심사 분리와 단일 책임 원칙을 찾아서

레벨 인터뷰를 진행하면서 단일 책임 원칙에 대한 질문을 받았었는데 명확하게 대답하지 못했습니다.
또 학습을 되돌아보니 UI와 도메인 로직을 분리하는 것을 핵심으로 배웠는데 명쾌하게 정리하지 못한다는 생각이 들었습니다.

그래서 방학기간동안 제 사라진 <b>관심사 분리</b>와 <b>단일 책임 원칙</b>을 되찾아보려고 했고, 제가 찾은 근거는 두 가지였습니다.

<ol>
  <li>한 문제에만 집중할 수 있다</li>
  <li>쉬운 대체 가능성</li>
</ol>

## 한 문제에만 집중한다는 것

관심사 분리와 단일 책임 원칙은 공통적으로 하나의 영역(객체)이 하나의 책임만을 가지도록 하는 원칙입니다. 이러한 원칙들은 문제를 더 작은 단위로 나누고 한 번에 집중해야 하는 범위를 줄이는 데 목적이 있습니다.

단일 책임 원칙과 관심사 분리를 통해 테스트 용이성, 교체 가능성 등 여러 부가적인 이점을 얻을 수 있습니다. 하지만 핵심적인 장점은 <b>복잡한 문제를 작게 쪼개고 특정 문제에 더 깊게 집중</b>할 수 있도록 해준다는 점이라고 생각합니다.

예를 들어 도메인 로직을 작성할 때는 데이터 접근이나 UI 로직을 신경 쓰지 않아도 되고, 할인율을 계산하는 객체를 수정할 때는 결제나 예약 같은 다른 영역을 고려하지 않아도 됩니다(단일 책임 원칙). 이렇게 관심사 분리가 되면 각 영역은 서로의 복잡도에 영향을 받지 않아 개발자는 각 문제를 독립적으로 생각하고 수정할 수 있게 됩니다.

## 대체하기 쉽다는 것

하나의 영역(객체)가 하나의 책임만을 가진다면 다른 구현으로 대체하기가 쉽습니다.

```kotlin
class MovieDiscount() {
  fun print() // UI 관심사
  fun apply() // 할인률 적용 관심사
}
```

위처럼 UI 출력과 할인 정책이 하나의 클래스에 함께 존재하면 각각의 변경이 서로에게 영향을 주게 됩니다.
만약 UI가 변경되면 apply에 대한 내용을 중복으로 가지는 새로운 클래스가 필요해지고, 새로운 할인 정책이 추가된다면 print 내용을 중복으로 가지는 새로운 클래스가 필요하게 될 것입니다. 이렇게 만들어진 중복은 조금만 많아져도 관리하기가 힘들어진다는 문제가 생깁니다.

그래서 객체가 한 가지 일만 하도록 하고 관심사를 분리하는 일을 통해 대체 가능성을 쉽게 높여(다형성) 기능 확장에 유연한 프로그램을 만들 수 있게 돕습니다.

```kotlin
interface Discountable {
  fun apply()
}

class MovieDiscount : Discountable
```
