# 25.05.18 월요일

**코틀린 코루틴의 정석**

- VM 옵션으로 `-Dkotlinx.coroutines.debug`를 사용하면 어떤 코루틴인지 확인이 쉽다.
- newSingleThreadContext 함수는 newFixedThreadPoolContext를 통해 만들어진다.
  - kotlinx.coroutine 1.7.2 기준으로는 구현체가 Executor 프레임워크를 사용한다.
- 스레드의 생성 비용은 비싸기에(동료가 존재를 모르고 다시 생성한다면?) 미리 정의해둔 목록이 있다.
  - Dispatchers.IO: 입출력 잡업은 IO Bound 작업으로 많은 스레드 필요
  - Dispatchers.Default: CPU Bound 작업
  - Dispatchers.Main
- CPU Bound 작업은 코루틴을 사용해도 스레드가 블록되기에 속도면에서 장점이 없다.
- Dispatchers.IO와 Dispatchers.Default는 스레드풀을 공유한다.
  - 왜 IO와 Default를 구분지은 것일까? → CPU Bound 작업을 위한 Default는 하드웨어 스레드/코어와 1:1 매칭을 시키는 것이 불필요한 컨텍스트 스위칭을 막을 수 있는 방법 아닐까?
- `runBlocking`, `launch`와 같은 함수는 코루틴 빌더 함수로 새로운 코루틴을 생성한다.
